1. 서비스 개요 및 기획 의도

1-1. 기획 배경

우리나라 교통약자(장애인, 고령자, 임산부 등)는 약 1,500만 명으로 전체 인구의 29%에 해당합니다. 「교통약자의 이동편의 증진법」 제3조는 교통약자가 안전하고 편리하게 이동할 수 있도록 교통수단, 여객시설 및 도로에 이동편의시설을 확충하도록 규정하고 있으나, 현실에서는 이동에 필요한 정보가 7개 이상의 서로 다른 시스템에 분산되어 통합 확인이 어려운 상황입니다.

1-2. 서비스 개요

"모두의 길"은 전국 7종 통합데이터(20개 API 엔드포인트)를 AI 대화형 인터페이스로 통합하여, 교통약자가 한 번의 질문으로 필요한 모든 이동 정보를 받아볼 수 있는 웹 서비스입니다.

1-3. 기획 의도
교통약자에게 "대화 한 마디로 안전한 길을 찾아주는 AI 동반자"를 제공합니다.

기존의 정보 조회 방식(검색 → 필터 → 결과 확인)을 자연어 대화 방식으로 전환하여, 디지털 기기에 익숙하지 않은 고령자도 쉽게 사용할 수 있도록 설계했습니다. 추천 질문 버튼 5개를 제공하여 첫 사용자도 별도 학습 없이 즉시 서비스를 경험할 수 있습니다.

3-3. 핵심 기술 구현

(A) 하이브리드 AI 아키텍처 — 서버 키워드 라우팅 + Claude NLG

본 서비스는 Claude의 tool_use(Function Calling)를 사용하지 않습니다. 대신, 속도와 AI 품질을 모두 확보하기 위해 2단계 하이브리드 아키텍처를 설계했습니다.

단계처리 주체동작장점
Stage 1서버 키워드 regex 라우팅사용자 질문에서 키워드 매칭 → 해당 도구를 Promise.all로 병렬 실행빠른 응답 (단일 Claude API 호출), 예측 가능한 도구 선택
Stage 2Claude Haiku 4.5도구 실행 결과를 user 메시지에 텍스트로 주입 → Claude가 맞춤형 자연어 응답 생성AI의 NLG 능력 활용, 교통약자 유형별 안내
이 설계의 핵심 이점:
  • Claude tool_use의 다중 턴 왕복(multi-turn round-trip)을 제거하여 응답 지연 최소화
  • 서버에서 도구를 직접 실행하므로 API 키가 클라이언트에 노출되지 않음
  • tools.ts에 7개 도구의 Claude Tool Use 스키마를 유지하여, 향후 자율적 도구 선택(autonomous tool selection)으로의 확장 경로 확보

처리 흐름 예시:

👤 사용자: "강남역 근처 신호등이랑 저상버스 알려줘"

"저상버스" → get_bus_realtime_location
"강남역" → region: "서울 강남구"
병렬 실행: 2개 API 동시 호출 (Promise.all)
▼
Stage 2: Claude NLG
도구 결과를 user 메시지에 주입 → Claude가 교통약자 맞춤 자연어 응답 생성
▼
🗺 지도 시각화

선언적 라우팅 설정 (route.ts):

const TOOL_ROUTING = [ { keywords: /도서관|빈자리|열람실|좌석/, tool: 'get_library_seats', alsoInclude: ['get_accessible_transport'] }, // 자동 연계: 도서관 → 교통약자 차량 { keywords: /민원실|민원|대기 현황/, tool: 'get_civil_office_wait', alsoInclude: ['get_accessible_transport'] }, // 자동 연계: 민원실 → 교통약자 차량 { keywords: /신호등|횡단보도/, tool: 'get_traffic_light_status' }, { keywords: /버스|저상버스/, tool: 'get_bus_realtime_location' }, { keywords: /자전거|따릉이|대여소/, tool: 'get_bicycle_availability' }, { keywords: /보관함|물품보관|짐 보관|락커/, tool: 'get_locker_availability' }, { keywords: /교통약자 차량|이동지원|특별교통/, tool: 'get_accessible_transport' }, ];
(B) 교통약자 유형별 맞춤 응답

Claude AI의 시스템 프롬프트에 교통약자 유형별 안내 지침을 설정하여, 동일한 데이터도 사용자 상황에 맞게 다르게 안내합니다.

사용자 유형맞춤 안내 내용
휠체어 사용자저상버스 위치, 경사로/엘리베이터 유무, 이동지원 차량 대기시간 우선 안내
시각장애인음향신호기 위치, 잔여시간 음성안내 여부, 점자블록 경로 안내
고령자충분한 신호 잔여시간, 가까운 시설, 대기 없는 서비스 우선 안내
임산부교통약자 전용 차량, 좌석 여유 있는 도서관/민원실 안내
(C) 데이터 자동 연계 (alsoInclude)

사용자가 단일 주제를 질문하더라도, 교통약자에게 필요한 관련 데이터를 자동으로 함께 조회합니다.

7-2. AI 활용 범위

(1) 자연어 이해 (NLU) — 서버 키워드 라우팅

  • 한국어 자연어 질문에서 regex 기반 키워드 매칭으로 필요한 도구 선택
  • 복합 질의 처리 ("신호등과 저상버스" → 2개 도구 동시 호출)
  • 지역명 문맥 이해 ("강남역 근처" → 서울 강남구, districtMap 매핑)

(2) 지능형 응답 생성 (NLG) — Claude AI

  • 7종 API의 원시 JSON 데이터를 맞춤형 자연어로 변환
  • 교통약자 유형별 맞춤 안내 (휠체어/시각장애/고령자/임산부)
  • 안전 주의사항 자동 강조 (잔여시간 부족 경고 등)
  • 추가 도움말 자동 제안 ("근처 저상버스 노선도 확인해볼까요?")

(3) 시스템 프롬프트 설계

  • 5가지 핵심 원칙 기반 행동 규칙 정의
  • 키워드 → 도구 매핑 규칙 명시
  • 교통약자 유형별 맞춤 안내 지침 내장
  • 응답 포맷 가이드라인 (표, 굵은 숫자, 경고 이모지)

7-2. (4) 하이브리드 아키텍처 설계 근거

방식장점단점채택 여부
Claude tool_use (Function Calling) Claude가 자율적으로 도구 선택 다중 턴 왕복으로 응답 지연 (2~3초 추가) 향후 확장용으로 스키마만 유지 (tools.ts)
서버 키워드 라우팅 + Claude NLG 단일 Claude 호출, 빠른 응답, 예측 가능 키워드 매칭 한계 현재 채택
현재 tools.ts에 7개 도구의 Claude Tool Use 스키마를 정의해 두어, 향후 사용자 질문이 복잡해지면 자율적 도구 선택으로 전환할 수 있는 확장 경로를 확보했습니다.
