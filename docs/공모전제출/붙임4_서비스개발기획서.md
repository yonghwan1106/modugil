# "모두의 길" — AI 기반 교통약자 통합 이동지원 서비스

---

## 1. 서비스 개요 및 기획 의도

### 1-1. 기획 배경

우리나라 교통약자(장애인, 고령자, 임산부 등)는 **약 1,500만 명으로 전체 인구의 29%** 에 해당합니다. 「교통약자의 이동편의 증진법」 제3조는 교통약자가 안전하고 편리하게 이동할 수 있도록 교통수단, 여객시설 및 도로에 이동편의시설을 확충하도록 규정하고 있으나, 현실에서는 이동에 필요한 정보가 7개 이상의 서로 다른 시스템에 분산되어 통합 확인이 어려운 상황입니다.

**현황 통계:**

| 지표 | 수치 | 출처 |
|------|------|------|
| 교통약자 인구 | 약 1,500만 명 (전체 인구 29%) | 국토교통부, 2025 |
| 저상버스 도입률 | 전국 42.3% | 국토교통부 교통약자 이동편의 실태조사 |
| 교통약자 이동지원 이용 대기시간 | 평균 30분 이상 | 한국장애인개발원 |

**핵심 문제:**

| 문제 | 현황 | 영향 |
|------|------|------|
| 횡단보도 신호 정보 부재 | 잔여시간 실시간 확인 불가 | 보행 안전 위협, 특히 고령자/지체장애인 |
| 교통수단 분산 | 저상버스/이동지원 차량 위치 파악 불가 | 불필요한 대기, 외출 포기 |
| 목적지 혼잡도 미확인 | 도서관 좌석, 민원실 대기시간 사전 확인 불가 | 장시간 대기로 체력 소진 |
| 정보 접근성 장벽 | 7개 이상 앱/사이트 번갈아 확인 | 디지털 취약 계층 정보 소외 |

### 1-2. 서비스 개요

**"모두의 길"** 은 전국 7종 통합데이터(20개 API 엔드포인트)를 AI 대화형 인터페이스로 통합하여, 교통약자가 **한 번의 질문**으로 필요한 모든 이동 정보를 받아볼 수 있는 웹 서비스입니다.

| 항목 | 내용 |
|------|------|
| **서비스명** | 모두의 길 (modugil) |
| **서비스 유형** | 반응형 웹 애플리케이션 (모바일/데스크톱) |
| **대상 사용자** | 교통약자 (장애인, 고령자, 임산부) 및 보호자 |
| **핵심 가치** | "하나의 질문, 모든 이동 정보" |
| **활용 데이터** | 전국 통합데이터 7종 20개 API 엔드포인트 연동, 이 중 14개를 AI 채팅 도구에서 실시간 활용, 나머지 6개는 데이터 프록시 API(`/api/data`)를 통해 대시보드 및 확장 기능에서 활용 |
| **GitHub** | https://github.com/yonghwan1106/modugil |
| **배포** | Vercel _(배포 후 기입)_ |

### 1-3. 기획 의도

> 교통약자에게 **"대화 한 마디로 안전한 길을 찾아주는 AI 동반자"** 를 제공합니다.

기존의 정보 조회 방식(검색 → 필터 → 결과 확인)을 **자연어 대화** 방식으로 전환하여, 디지털 기기에 익숙하지 않은 고령자도 쉽게 사용할 수 있도록 설계했습니다. 추천 질문 버튼 5개를 제공하여 첫 사용자도 별도 학습 없이 즉시 서비스를 경험할 수 있습니다.

### 3-3. 핵심 기술 구현

#### (A) 하이브리드 AI 아키텍처 — 서버 키워드 라우팅 + Claude NLG

본 서비스는 **Claude의 tool_use(Function Calling)를 사용하지 않습니다.** 대신, 속도와 AI 품질을 모두 확보하기 위해 2단계 하이브리드 아키텍처를 설계했습니다.

| 단계 | 처리 주체 | 동작 | 장점 |
|------|----------|------|------|
| **Stage 1** | 서버 키워드 regex 라우팅 | 사용자 질문에서 키워드 매칭 → 해당 도구를 `Promise.all`로 **병렬 실행** | 빠른 응답 (단일 Claude API 호출), 예측 가능한 도구 선택 |
| **Stage 2** | Claude Haiku 4.5 | 도구 실행 결과를 user 메시지에 텍스트로 주입 → Claude가 **맞춤형 자연어 응답** 생성 | AI의 NLG 능력 활용, 교통약자 유형별 안내 |

**이 설계의 핵심 이점:**
- Claude tool_use의 다중 턴 왕복(multi-turn round-trip)을 제거하여 **응답 지연 최소화**
- 서버에서 도구를 직접 실행하므로 **API 키가 클라이언트에 노출되지 않음**
- `tools.ts`에 7개 도구의 Claude Tool Use 스키마를 유지하여, 향후 자율적 도구 선택(autonomous tool selection)으로의 **확장 경로 확보**

**처리 흐름 예시:**
```
사용자: "강남역 근처 신호등이랑 저상버스 알려줘"
  ↓
[Stage 1: 서버 키워드 라우팅]
  키워드 분석: "신호등" → get_traffic_light_status
               "저상버스" → get_bus_realtime_location
               "강남역" → region: "서울 강남구"
  병렬 실행: 2개 API 동시 호출 (Promise.all)
  ↓
[Stage 2: Claude NLG]
  도구 결과를 user 메시지에 주입 → Claude가 교통약자 맞춤 자연어 응답 생성
  ↓
지도 시각화: 신호등(노랑 마커) + 버스(초록 마커) 동시 표시
```

**선언적 라우팅 설정 (route.ts):**
```typescript
const TOOL_ROUTING = [
  { keywords: /도서관|빈자리|열람실|좌석/, tool: 'get_library_seats',
    alsoInclude: ['get_accessible_transport'] },  // 자동 연계: 도서관 → 교통약자 차량
  { keywords: /민원실|민원|대기 현황/, tool: 'get_civil_office_wait',
    alsoInclude: ['get_accessible_transport'] },   // 자동 연계: 민원실 → 교통약자 차량
  { keywords: /신호등|횡단보도/, tool: 'get_traffic_light_status' },
  { keywords: /버스|저상버스/, tool: 'get_bus_realtime_location' },
  { keywords: /자전거|따릉이|대여소/, tool: 'get_bicycle_availability' },
  { keywords: /보관함|물품보관|짐 보관|락커/, tool: 'get_locker_availability' },
  { keywords: /교통약자 차량|이동지원|특별교통/, tool: 'get_accessible_transport' },
];
```

#### (B) 교통약자 유형별 맞춤 응답

Claude AI의 시스템 프롬프트에 교통약자 유형별 안내 지침을 설정하여, 동일한 데이터도 사용자 상황에 맞게 다르게 안내합니다.

| 사용자 유형 | 맞춤 안내 내용 |
|------------|---------------|
| 휠체어 사용자 | 저상버스 위치, 경사로/엘리베이터 유무, 이동지원 차량 대기시간 우선 안내 |
| 시각장애인 | 음향신호기 위치, 잔여시간 음성안내 여부, 점자블록 경로 안내 |
| 고령자 | 충분한 신호 잔여시간, 가까운 시설, 대기 없는 서비스 우선 안내 |
| 임산부 | 교통약자 전용 차량, 좌석 여유 있는 도서관/민원실 안내 |

#### (C) 데이터 자동 연계 (alsoInclude)

사용자가 단일 주제를 질문하더라도, 교통약자에게 필요한 관련 데이터를 **자동으로 함께 조회**합니다.

| 사용자 질문 | 직접 호출 도구 | 자동 연계 도구 | 이유 |
|------------|--------------|--------------|------|
| "종로구 도서관 빈자리" | get_library_seats | get_accessible_transport | 도서관까지 이동수단 안내 |
| "민원실 대기 현황" | get_civil_office_wait | get_accessible_transport | 민원실까지 이동수단 안내 |

### 3-4. 안정성 및 성능 설계

| 설계 항목 | 구현 방식 | 효과 |
|----------|----------|------|
| **인메모리 캐시** | 5분 TTL 캐시 (client.ts) | 동일 요청 반복 시 API 호출 절감 |
| **Mock 데이터 폴백** | API 장애 시 자동 전환 (fetchWithFallback) | 서비스 중단 없이 시연 가능 |
| **타임아웃 + 재시도** | 30초 타임아웃, 자동 1회 재시도 (chatWithRetry) | 네트워크 불안정 대응 |
| **병렬 실행** | Promise.all로 복수 API 동시 호출 | 응답 속도 최적화 (직렬 대비 2~3배) |
| **에러 UX** | 에러 메시지 + "다시 시도" 버튼 (handleRetry) | 마지막 질문 자동 재전송으로 사용자 편의 |
| **API 키 서버사이드 관리** | 환경변수로 서버에서만 사용 | 클라이언트 노출 방지, 보안 확보 |

### 5-1. 크로스 데이터 통합 활용

"모두의 길"의 핵심 가치는 **7종 데이터의 교차 활용**입니다. 단일 데이터 조회를 넘어, 교통약자의 실제 외출 시나리오에 맞춰 복수 데이터를 자동으로 통합 제공합니다.

| 활용 시나리오 | 통합 데이터 | 자동 연계 (alsoInclude) | 사용자 가치 |
|-------------|-----------|----------------------|-----------|

## 7. AI 활용 상세 (가산점 10점)

### 7-1. AI 모델 정보

| 항목 | 내용 |
|------|------|
| **모델** | Claude Haiku 4.5 (Anthropic) |
| **SDK** | @anthropic-ai/sdk v0.80 |
| **활용 방식** | 서버사이드 API 호출 (Next.js API Route) |
| **호출 구조** | 하이브리드 — 서버 키워드 라우팅(Stage 1) + Claude NLG(Stage 2) |

### 7-2. AI 활용 범위

#### (1) 자연어 이해 (NLU) — 서버 키워드 라우팅

- 사용자의 한국어 자연어 질문에서 **regex 기반 키워드 매칭**으로 필요한 도구 선택
- 복합 질의 처리 ("신호등**과** 저상버스" → 2개 도구 동시 호출)
- 지역명 문맥 이해 ("강남역 근처" → 서울 강남구, districtMap 매핑)

#### (2) 지능형 응답 생성 (NLG) — Claude AI

- 7종 API의 원시 JSON 데이터를 사람이 읽기 쉬운 **맞춤형 자연어**로 변환
- 교통약자 유형별 맞춤 안내 (휠체어/시각장애/고령자/임산부)
- 안전 주의사항 자동 강조 (잔여시간 부족 경고 등)
- 추가 도움말 자동 제안 ("근처 저상버스 노선도 확인해볼까요?")

#### (3) 시스템 프롬프트 설계

- 5가지 핵심 원칙 기반 행동 규칙 정의
- 키워드 → 도구 매핑 규칙 명시
- 교통약자 유형별 맞춤 안내 지침 내장
- 응답 포맷 가이드라인 (표, 굵은 숫자, 경고 이모지)

#### (4) 하이브리드 아키텍처 설계 근거

| 방식 | 장점 | 단점 | 채택 여부 |
|------|------|------|----------|
| **Claude tool_use (Function Calling)** | Claude가 자율적으로 도구 선택 | 다중 턴 왕복으로 응답 지연 (2~3초 추가) | 향후 확장용으로 스키마만 유지 (tools.ts) |
| **서버 키워드 라우팅 + Claude NLG** | 단일 Claude 호출, 빠른 응답, 예측 가능 | 키워드 매칭 한계 | **현재 채택** |

현재 `tools.ts`에 7개 도구의 Claude Tool Use 스키마를 정의해 두어, 향후 사용자 질문이 복잡해지면 자율적 도구 선택으로 전환할 수 있는 **확장 경로를 확보**했습니다.

### 7-3. AI가 서비스에 기여하는 핵심 가치

| 기존 방식 (AI 미적용) | 모두의 길 (AI 적용) |
|---------------------|-------------------|
| 7개 앱/사이트 각각 방문 | **한 번의 대화**로 통합 조회 |
| 원시 JSON 데이터 직접 해석 | AI가 **맞춤형 자연어**로 안내 |
| 일반적인 정보 제공 | **교통약자 유형별** 맞춤 안전 안내 |
| 단일 데이터 조회 | **복합 질의** 지원 + 자동 연계(alsoInclude) |
| 수동 위험 판단 | AI가 **위험 상황 자동 경고** |

### 7-4. 시스템 프롬프트 발췌 (system-prompt.ts)

아래는 실제 코드에서 사용 중인 시스템 프롬프트의 핵심 부분입니다.

**핵심 원칙 5가지:**

```
1. 항상 도구를 사용하여 실시간 데이터를 조회한 후 답변합니다. 도구 없이 절대 답변하지 않습니다.
2. 아래 라우팅 표에 따라 첫 번째 응답에서 해당하는 모든 도구를 한 번에 동시 호출합니다.
3. 데이터는 표/목록 형태로 보기 좋게 정리합니다.
4. 위치 데이터(위도/경도)가 포함된 결과는 반드시 "지도에 위치를 표시했습니다"라고 안내합니다.
5. 데이터가 없는 경우 솔직히 안내하고 대안을 제시합니다.
```

**교통약자 유형별 맞춤 안내 지침:**

```
- 휠체어 사용자: 저상버스 위치, 경사로/엘리베이터 유무, 교통약자 차량 대기시간 우선 안내
- 시각장애인: 음향신호기 위치, 잔여시간 음성 안내 여부, 점자블록 경로 안내
- 고령자: 신호 잔여시간이 충분한 횡단보도, 가까운 시설, 대기 없는 서비스 우선 안내
- 임산부: 교통약자 전용 차량, 좌석 여유 있는 도서관/민원실 안내
```
